# タスク管理アプリ 要件定義書

**バージョン**: 0.1（ドラフト）
**作成日**: 2026-05-17
**ステータス**: レビュー中

---

## 1. プロジェクト概要

### 背景・課題

仕事上のタスクを口頭やメモで管理しているため、タスク漏れが発生しやすい。既存のツール（Trello等）は多機能すぎて個人利用には重く、日常的に使い続けるハードルが高い。

### 目的

20代社会人が日々のタスクを抜け漏れなく管理できる、シンプルなWebアプリを構築する。将来的にはLINEチャットボットからタスク登録・管理・リマインドを行えるようにし、ツールを開かなくてもタスク管理できる状態を目指す。

### デモ版のスコープ

- Webブラウザ上でのタスク管理（CRUD・ステータス管理・依存関係）
- メール通知（タスク操作・リマインド）
- 将来のLINE連携を見越したアダプター設計

---

## 2. ターゲットユーザー

### ペルソナ

| 項目 | 内容 |
|---|---|
| 年齢 | 20代 |
| 職種 | 会社員（業種不問） |
| 課題 | タスクが増えると抜け漏れが発生し、後から気づいて焦ることがある |
| 行動特性 | LINEは毎日使うが、タスク管理ツールは続かない |

### 解決したい状態

- タスクを一元管理できている
- 期日が近いタスクを自動で通知してもらえる
- 依存関係があるタスクの順序を可視化できる

## 3. 機能要件

### 3.1 タスク管理

| # | 機能 | 説明 |
|---|---|---|
| F-01 | タスク作成 | タイトル・期日・カテゴリ・優先度（高/中/低、デフォルト:中）・メモ・依存関係（FS/FF）を入力して新規作成 |
| F-02 | タスク一覧表示 | ステータス（未着手／進行中／完了）でフィルタリングして一覧表示 |
| F-03 | タスク詳細表示 | 全フィールドを確認できる詳細画面 |
| F-04 | タスク編集 | 各フィールドを更新できる編集画面 |
| F-05 | タスク削除 | タスクを削除する（物理削除）。他のタスクから依存されている場合は削除不可とし、エラーを表示する |
| F-06 | ステータス変更 | 未着手 → 進行中 → 完了 の一方向ワークフローで更新（差し戻し不可） |

### 3.2 カテゴリ管理

| # | 機能 | 説明 |
|---|---|---|
| F-07 | カテゴリ管理 | カテゴリの追加・編集・削除 |

### 3.3 依存関係

| # | 機能 | 説明 |
|---|---|---|
| F-08 | 依存関係設定 | タスク作成・編集時に他タスクとの依存関係を紐付け。種別は2種類：**FS（Finish-to-Start）**＝前タスクが完了するまで「進行中」にできない、**FF（Finish-to-Finish）**＝前タスクが完了するまで「完了」にできない |
| F-09 | 依存関係の可視化 | 依存タスクをバッジまたはリンクで表示 |
| F-10 | 依存違反のブロック | FS依存：前タスクが未完了の状態で「進行中」への変更をブロックしエラーを表示。FF依存：前タスクが未完了の状態で「完了」への変更をブロックしエラーを表示 |

### 3.4 検索・表示補助

| # | 機能 | 説明 |
|---|---|---|
| F-11 | 検索・フィルター | カテゴリ・期日・ステータスで絞り込み |
| F-12 | ソート | 期日順・作成日順で並び替え |
| F-13 | 期日超過ハイライト | 期日を過ぎた未完了タスクを赤色などで視覚的に強調表示 |

### 3.5 リマインド

| # | 機能 | 説明 |
|---|---|---|
| F-14 | リマインド設定 | タスクごとに「X日前」または「X時間前」の通知タイミングを1件単位で設定。1タスクに複数のリマインドを登録可能（例：3日前と1時間前をそれぞれ別レコードで登録） |
| F-15 | リマインド一覧・編集 | 設定済みリマインドの確認・変更・削除 |

### 3.6 ユーザー設定

| # | 機能 | 説明 |
|---|---|---|
| F-19 | 通知先メールアドレス設定 | ユーザーが通知を受け取るメールアドレスをアプリ内で設定・変更できる |

### 3.7 通知・外部連携

| # | 機能 | 説明 |
|---|---|---|
| F-16 | メール通知（デモ版） | タスク追加・削除・完了・リマインド発火のタイミングでメールを送信。送信サービスは **Resend**（無料枠）を使用 |
| F-17 | 通知送信モジュール（アダプター設計） | `NotificationService` インターフェースを定義し、`EmailAdapter` を実装。将来 `LineAdapter` を追加するだけでLINE通知に対応できる設計 |
| F-18 | Webhook受信エンドポイント | LINE Bot等からのリクエストを受け付けるエンドポイントを用意（デモ版では受信ログのみ） |

**通知が発火するイベント一覧**

| イベント | 通知内容（例） |
|---|---|
| タスク追加 | 「タスク『〇〇』を登録しました（期日：〇月〇日）」 |
| タスク削除 | 「タスク『〇〇』を削除しました」 |
| タスク完了 | 「タスク『〇〇』が完了しました」 |
| リマインド | 「タスク『〇〇』の期日まであとX日です」 |

## 4. データモデル

### 4.1 ER図（概要）

```
user_settings
      |
      | (通知先)
      |
categories ──── tasks ──── task_dependencies
                  │
                  ├── reminders
                  └── webhook_logs（将来用）
```

### 4.2 テーブル定義

#### `tasks`（タスク）

| カラム名 | 型 | 必須 | 説明 |
|---|---|---|---|
| id | UUID | ✓ | PK |
| title | VARCHAR(255) | ✓ | タスク名 |
| memo | TEXT | | メモ |
| status | ENUM | ✓ | `未着手` / `進行中` / `完了` |
| priority | ENUM | ✓ | `高` / `中` / `低` |
| due_date | TIMESTAMPTZ | | 期日 |
| completed_at | TIMESTAMPTZ | | 完了日時（完了時に自動セット） |
| category_id | UUID | | FK → categories.id |
| created_at | TIMESTAMPTZ | ✓ | 作成日時 |
| updated_at | TIMESTAMPTZ | ✓ | 更新日時 |

#### `categories`（カテゴリ）

| カラム名 | 型 | 必須 | 説明 |
|---|---|---|---|
| id | UUID | ✓ | PK |
| name | VARCHAR(100) | ✓ | カテゴリ名 |
| color | VARCHAR(7) | | カラーコード（例：`#FF5733`） |
| created_at | TIMESTAMPTZ | ✓ | 作成日時 |

#### `task_dependencies`（タスク依存関係）

| カラム名 | 型 | 必須 | 説明 |
|---|---|---|---|
| id | UUID | ✓ | PK |
| task_id | UUID | ✓ | FK → tasks.id（依存する側のタスク） |
| depends_on_task_id | UUID | ✓ | FK → tasks.id（前提となるタスク） |
| dependency_type | ENUM | ✓ | `FS`（前タスク完了後に開始）/ `FF`（同時完了） |

#### `reminders`（リマインド設定）

| カラム名 | 型 | 必須 | 説明 |
|---|---|---|---|
| id | UUID | ✓ | PK |
| task_id | UUID | ✓ | FK → tasks.id |
| notify_before_days | INTEGER | | 期日の何日前に通知（0以上） |
| notify_before_hours | INTEGER | | 期日の何時間前に通知（0以上） |
| scheduled_at | TIMESTAMPTZ | ✓ | 通知予定日時（due_date - 設定値で算出） |
| is_sent | BOOLEAN | ✓ | 送信済みフラグ（重複送信防止）デフォルト: false |

#### `user_settings`（ユーザー設定）

| カラム名 | 型 | 必須 | 説明 |
|---|---|---|---|
| id | UUID | ✓ | PK（常に1レコード） |
| notification_email | VARCHAR(255) | | 通知先メールアドレス |
| updated_at | TIMESTAMPTZ | ✓ | 更新日時 |

#### `webhook_logs`（Webhookログ・将来用）

| カラム名 | 型 | 必須 | 説明 |
|---|---|---|---|
| id | UUID | ✓ | PK |
| source | VARCHAR(50) | ✓ | 送信元（例：`line`） |
| payload | JSONB | ✓ | 受信したリクエストボディ |
| received_at | TIMESTAMPTZ | ✓ | 受信日時 |

## 5. 外部連携設計

### 5.1 通知アーキテクチャ

アダプターパターンで実装し、将来のLINE対応をコード変更最小で行えるようにする。

```
通知イベント発火
      ↓
NotificationService（共通インターフェース）
      ↓
┌─────────────────────┐
│ EmailAdapter        │  ← デモ版で実装（Resend API）
│ LineAdapter（stub） │  ← 将来実装（メソッドのみ定義）
└─────────────────────┘
```

### 5.2 通知イベント定義

| イベント定数 | 発火タイミング | メール本文（例） |
|---|---|---|
| `TASK_CREATED` | タスク作成時 | 「タスク『〇〇』を登録しました（期日：〇月〇日）」 |
| `TASK_DELETED` | タスク削除時 | 「タスク『〇〇』を削除しました」 |
| `TASK_COMPLETED` | ステータスを完了に変更時 | 「タスク『〇〇』が完了しました」 |
| `REMINDER` | リマインド送信バッチ実行時 | 「タスク『〇〇』の期日まであとX日X時間です」 |

### 5.3 メール送信（EmailAdapter / Resend）

- 送信サービス：**Resend**（無料枠：月3,000通）
- 送信元アドレス：`onboarding@resend.dev`（共有ドメイン、アカウント登録のみで即日利用可）
- 宛先：`user_settings.notification_email` に設定されたアドレス
- APIキーは環境変数 `RESEND_API_KEY` で管理（コードに直書きしない）

### 5.4 リマインドスケジューラー

定期バッチ（1分間隔）で以下を実行する。

```
SELECT * FROM reminders
WHERE scheduled_at <= NOW()
  AND is_sent = false
```

取得したレコードに対して `NotificationService.send(REMINDER, task)` を呼び出し、成功後に `is_sent = true` に更新する。

- **リトライ**：Resend APIがエラーを返した場合、最大3回リトライする。3回失敗した場合はエラーログに記録し、`is_sent` は `false` のまま維持（次回バッチで再試行）。

### 5.5 Webhook受信エンドポイント

- エンドポイント：`POST /api/webhooks/line`
- デモ版の動作：受信したペイロードを `webhook_logs` テーブルに保存してログとして記録するのみ
- 将来：LINEのメッセージ内容を解析してタスク操作に変換する処理を追加

### 5.6 環境変数一覧

| 変数名 | 説明 |
|---|---|
| `RESEND_API_KEY` | Resend APIキー |
| `DATABASE_URL` | PostgreSQL接続文字列（Render提供） |
| `REMINDER_BATCH_INTERVAL` | リマインドバッチの実行間隔（秒、デフォルト60） |

## 6. 非機能要件

| 分類 | 要件 |
|---|---|
| パフォーマンス | 一覧・詳細画面のレスポンスは2秒以内 |
| セキュリティ | APIキー・DB接続文字列は環境変数で管理し、コードに直書きしない |
| 通信 | HTTPS必須（Renderが自動付与） |
| 可用性 | デモ用途のため99%以上の稼働は保証しない（Renderの無料枠を許容） |
| ブラウザ対応 | Chrome最新版を基準とする |

## 7. 技術スタック

### バックエンド

| 技術 | 選定理由 |
|---|---|
| Python 3.12 + FastAPI | 型ヒント標準対応・OpenAPI自動生成・非同期処理対応 |
| SQLAlchemy + Alembic | ORMとマイグレーション管理の標準的な組み合わせ |
| APScheduler | リマインドバッチの定期実行（プロセス内で動作、外部依存なし） |
| Resend Python SDK | メール送信 |
| PostgreSQL（Render） | 本番相当のRDBMS・Renderに無料枠あり |

### フロントエンド

| 技術 | 選定理由 |
|---|---|
| React + TypeScript | 型安全・コンポーネント再利用性 |
| Vite | 高速ビルド |
| TanStack Query | サーバー状態管理・キャッシュ |
| Tailwind CSS | ユーティリティベースで素早くスタイリング |

### インフラ

| 技術 | 用途 |
|---|---|
| Render（Web Service） | バックエンドホスティング |
| Render（PostgreSQL） | データベース |
| Render（Static Site） | フロントエンドホスティング |

## 8. スコープ外

以下は今回のデモ版には含まない。

| 項目 | 理由 |
|---|---|
| 他ユーザーへのタスク共有 | 個人利用に特化するため |
| LINEチャットボットの本実装 | 将来フェーズ。デモ版はWebhook受信ログのみ |
| モバイルアプリ（iOS / Android） | Webアプリで代替 |
| タスクのインポート / エクスポート | 優先度が低いため |
| 認証・ログイン機能 | デモ版は単一ユーザー想定のため不要 |
| メール以外の通知チャネル（Slack等） | LINEを最初の拡張先と定め、それ以外は対象外 |
